# Bug Report — School Visit Tracker

Findings from a static review of the Next.js frontend and Google Apps Script backend. No code was changed as part of this report.

| # | Bug | File | Line | Severity | Suggested Fix |
|---|-----|------|------|----------|----------------|
| 1 | GPS fields captured by the frontend (`latitude`, `longitude`, `accuracy`) are silently dropped — `VISIT_FIELDS` only lists `google_map`, so `handleAddVisit` never writes those three values to the sheet even though the UI requires them before allowing submission. | `google-apps-script/Code.gs` | 34–51 (list), 151–163 (`handleAddVisit`) | High | Add `"latitude"`, `"longitude"`, `"accuracy"` to `VISIT_FIELDS` and redeploy the Apps Script. |
| 2 | Passwords are stored and compared in plaintext (`Users` sheet, `handleLogin`). Anyone with edit access to the sheet (or a leaked `/exec` URL response) sees credentials in the clear. | `google-apps-script/Code.gs` | 75, 124–146 | High | Hash passwords (e.g. `Utilities.computeDigest`) before storing and compare hashes; never persist raw passwords. |
| 3 | User-supplied text (school name, notes, report, etc.) is written directly into sheet cells via `appendRow` with no sanitization, allowing spreadsheet formula/CSV injection (e.g. a value starting with `=`, `+`, `-`, or `@`). | `google-apps-script/Code.gs` | 151–163 (`handleAddVisit`) | Medium | Escape/prefix values that start with formula-trigger characters (e.g. prepend `'`) before writing to the sheet. |
| 4 | "Today" is computed with `new Date().toISOString().slice(0,10)`, which is UTC-based. For users in non-UTC timezones this rolls over at the wrong local time, so visits near midnight can be misfiled under the wrong date / excluded from "Today's Total Visits". | `app/new-visit/page.tsx` (todayISO), `app/dashboard/page.tsx` (todayISO) | 19–21, 27–30 | Medium | Build the date string from local `getFullYear()/getMonth()/getDate()` instead of `toISOString()`. |
| 5 | "My Submitted Visits" is matched by comparing the free-text `executive` field (editable on the New Visit form) against the logged-in user's `name`, case-insensitively but without trimming whitespace. Any retyping, extra spaces, or nickname variance causes a submitted visit to disappear from the user's own list. | `app/dashboard/page.tsx` (filter), `app/new-visit/page.tsx` (editable executive input) | 73; 26, 123–128 | Medium | Match on the stable `username` instead of a free-typed display name, or make the executive field read-only / derived from the session. |
| 6 | `doPost` silently defaults any missing/unrecognized `action` to `"addVisit"` instead of rejecting it, so a malformed request can append a garbage row to the `Visits` sheet rather than surfacing an error. | `google-apps-script/Code.gs` | 108 | Medium | Return an explicit error response when `action` is missing or unrecognized instead of defaulting to `addVisit`. |
| 7 | No throttling/lockout on login attempts — `handleLogin` allows unlimited password guesses against the `Users` sheet. | `google-apps-script/Code.gs` | 124–146 | Medium | Add attempt counting/lockout (e.g. via `PropertiesService` or `CacheService`) per username/IP. |
| 8 | `handleLogin` trims and lowercases `username` before comparing but does not trim `password`, so a stray trailing space in the sheet cell (or pasted password) causes an otherwise-correct login to fail with a generic error. | `google-apps-script/Code.gs` | 139 | Low | Trim both sides before comparing: `String(rowPass).trim() === String(password).trim()`. |
| 9 | Visit list items are keyed by array index (`key={idx}`) rather than a stable identifier. Since the list is re-fetched/reordered on refresh, React can misassociate component state (e.g., during animations) with the wrong row. | `app/dashboard/page.tsx` | 136 | Low | Key on a stable field, e.g. `` `${v.timestamp}-${v.school_name}` ``, or have the backend return a row id. |
| 10 | When `v.interest` is empty/unrecognized (e.g. legacy rows, or a value that doesn't exactly match `Hot`/`Warm`/`Cold`), the UI silently falls back to the "Warm" badge style while still printing the empty/invalid text, showing a misleading amber badge with no label. | `app/dashboard/page.tsx` | 133, 153 | Low | Render an explicit "Unknown" state (or omit the badge) when `interestBadge[v.interest]` has no match, instead of defaulting to `Warm`. |
| 11 | If `NEXT_PUBLIC_APPS_SCRIPT_URL` is unset, the app only logs a `console.warn` and continues; `apiClient` ends up posting to `baseURL: ""` (a relative path against the Next.js app itself), producing a confusing generic "Could not reach the server" toast instead of a clear configuration error. | `lib/api.ts` | 9–16, 26–32 | Low | Fail fast with a clear, user-visible configuration error when the env var is missing, instead of only warning to the console. |
| 12 | Form field ids are auto-derived from the label text (`label.toLowerCase().replace(/\s+/g, "-")`) with no collision check. Two fields sharing a label on the same page would produce duplicate DOM `id`s, breaking `<label htmlFor>` association for assistive tech. Not currently triggered by any existing page, but the pattern is unguarded. | `components/Input.tsx`, `components/Textarea.tsx`, `components/Select.tsx` | 10, 9, 17 | Low | Generate ids from a required explicit `name`/`id` prop, or add a uniqueness guard (e.g. `useId()`). |

## Notes
- Severity reflects impact within this app's stated design (single Google Sheet backend, small internal user base), not a general production security posture.
- Bug #1 is the most impactful: the location-capture feature is presented as mandatory for submission, but its core output (coordinates) is never actually saved.
